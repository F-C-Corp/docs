## API scope

<Warning>
  The OpenFiskal API currently supports server-to-server communication only.
  Do not embed API keys in POS devices, mobile apps, browsers, or other client-side surfaces.
</Warning>

## API key creation

Create API keys in the tenant dashboard. Each key is tenant-scoped and country-scoped.

<Warning>
  Store the key in your secrets manager. Treat it like any other production secret.
</Warning>

## Use the standard authorization header

Although the credential is an API key, the API uses the standard HTTP bearer pattern:

```bash
-H "Authorization: Bearer ofk_live_abc123..."
```

This keeps the API compatible with standard API tooling, gateways, proxies, and security middleware.

## Organization scoping

For every organization-scoped request, also send:

```bash
-H "X-OpenFiskal-Organization: org_01HXYZ"
```

This header must contain the OpenFiskal organization ID, not the API key.

Together, a typical organization-scoped request looks like this:

```bash
curl https://api.openfiskal.com/v1/operations \
  -H "Authorization: Bearer ofk_live_abc123..." \
  -H "X-OpenFiskal-Organization: org_01HXYZ"
```

Organization-scoped resources include:

- Locations
- Registers
- Operations
- Exports
- Receipts

You do not need `X-OpenFiskal-Organization` when you create, update, delete, or read the organization itself.

## Idempotency

For mutating requests that create or finalize state, send an `Idempotency-Key` header so retries remain safe:

```bash
curl -X POST https://api.openfiskal.com/v1/operations \
  -H "Authorization: Bearer ofk_live_abc123..." \
  -H "X-OpenFiskal-Organization: org_01HXYZ" \
  -H "Idempotency-Key: op-order-1001-start" \
  -H "Content-Type: application/json" \
  -d '{ "...": "..." }'
```

Use idempotency keys for operation start, operation completion, operation cancellation, export creation, and receipt delivery.

The platform retains keys for at least 24 hours. Reusing the same key with a different payload returns `409 idempotency_key_conflict`.

## Conditional writes

Open operation mutations use server-issued resource versions.

1. Read or create the operation.
2. Persist the returned `ETag` and `resource_version`.
3. Send that value in `If-Match` on `PATCH /operations/{id}`, `PATCH /operations/{id}/complete`, and `PATCH /operations/{id}/cancel`.
4. Replace your stored value with the new `ETag` and `resource_version` from every successful mutation response.

The server issues a new operation resource version on every successful mutation. The response body returns that value as `resource_version`, and the response header returns the same version as `ETag`.

If you omit `If-Match`, the API returns `428 precondition_required`. If you send a stale value, the API returns `412 precondition_failed`.

## Recommended handling

- Return `401` when authentication is missing or invalid.
- Return `403` when the API key is valid but not allowed to access the requested tenant, country, or organization.
- Return `412 Precondition Failed` when an operation update, completion, or cancellation uses a stale `If-Match` value.
- Return `429 Too Many Requests` with `Retry-After` when the caller exceeds `500 requests/second` per API key.
