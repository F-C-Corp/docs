## Overview

Every fiscal event that occurs at a register is reported to OpenFiskal as an operation. Your backend sends the event, OpenFiskal applies the fiscal rules for the target regime, and the response carries the fiscal payload you need for receipts, exports, and auditability.

## Lifecycle

Each operation follows the same public lifecycle:

```mermaid
stateDiagram-v2
    [*] --> open
    open --> pending_payment
    pending_payment --> completed
    pending_payment --> payment_pending
    payment_pending --> completed
    open --> cancelled
    pending_payment --> cancelled
```

- `open` means the basket or order is still being assembled
- `pending_payment` means the financial intent is known and the operation is waiting for settlement input
- `payment_pending` means the operation has been completed with an asynchronous payment state that still needs follow-up
- `completed` means the fiscal event is final
- `cancelled` means the operation was intentionally voided before completion

Use a dedicated cancel action on the existing operation resource. Do not model cancellation as a separate operation type.

## Operation types

The public API currently supports these operation types:

| Type | When to use it |
| --- | --- |
| `sale` | Standard sale flow |
| `refund` | Return against an earlier sale |
| `exchange` | Return plus replacement in a single operation |
| `custom` | Platform-specific fiscal event that still needs to be recorded |

## Start an operation

Create operations with `POST /operations`. The request includes a `source` field with `POS` or `ONLINE`, and the `data` payload is a typed operation variant such as `SaleOperation` or `RefundOperation`. The response returns the operation resource, a `resource_version`, and an `ETag` header.

```bash
curl -X POST https://api.openfiskal.com/v1/operations \
  -H "Authorization: Bearer $API_KEY" \
  -H "X-OpenFiskal-Organization: org_01HXYZ" \
  -H "Idempotency-Key: op-order-1001-start" \
  -H "Content-Type: application/json" \
  -d '{ "...": "..." }'
```

## Update an open operation

Use `PATCH /operations/{operationId}` while the operation is still mutable. Send the latest `ETag` value in `If-Match`.

This makes retries and concurrent writers safe:

- the server owns the version
- every successful mutation returns a new `resource_version` and `ETag`
- the client retries with the same idempotency key where supported
- stale writers receive `412 precondition_failed`

## Complete an operation

Use `PATCH /operations/{operationId}/complete` when you want OpenFiskal to produce the final fiscal document and receipt state. Completion now accepts a typed `payments` array and can return `fiscal_information` for regime-specific fiscal payloads.

Completion is a fiscal event. Use an idempotency key on every completion request.

## Cancel an open operation

Use `PATCH /operations/{operationId}/cancel` only while the operation is still open or pending payment.

```bash
curl -X PATCH https://api.openfiskal.com/v1/operations/op_01HXYZ/cancel \
  -H "Authorization: Bearer $API_KEY" \
  -H "X-OpenFiskal-Organization: org_01HXYZ" \
  -H "Idempotency-Key: op-order-1001-cancel" \
  -H 'If-Match: "opv_01HXYZ"' \
  -H "Content-Type: application/json" \
  -d '{
    "reason": "void_before_completion"
  }'
```

Completed operations remain completed. This cancel action is not a replacement for refund or reversal workflows.

## Concurrency model

Use conditional writes with server-issued versions:

- create or read the operation
- persist the returned `ETag`
- treat `resource_version` in the body and `ETag` in the header as the same server-issued version
- send that value in `If-Match` on the next mutation
- replace your stored version after every successful mutation
- re-read on `412 precondition_failed`

## What your backend must do

- report fiscal events in order
- preserve idempotency keys for retries
- persist the latest `ETag` for every mutable operation
- print or deliver the returned fiscal payload exactly as provided
- complete or cancel every operation that you start
