## Overview

Each payment entry represents one authorization, cash leg, voucher redemption, or refund settlement leg.

## Payment enums

| Field | Allowed values |
| --- | --- |
| `method` | `cash`, `card`, `voucher`, `gift_card`, `bank_transfer`, `digital_wallet`, `store_credit`, `other` |
| `status` | `pending`, `authorized`, `partially_approved`, `captured`, `failed`, `voided`, `refunded` |

## Split tender

Use one payment object per tender leg. The sum of all `captured` and `authorized` amounts must match the operation total after any partial approval handling and any operator-approved remainder.

```json
{
  "payments": [
    {
      "payment_id": "pay_cash_1001",
      "method": "cash",
      "status": "captured",
      "amount": 1500,
      "currency": "EUR"
    },
    {
      "payment_id": "pay_card_1001",
      "method": "card",
      "status": "captured",
      "amount": 3250,
      "currency": "EUR",
      "processor": "sumup",
      "processor_reference": "ch_123"
    }
  ]
}
```

## Partial approvals

If the acquirer approves less than the requested amount, report `status: partially_approved` and include both `requested_amount` and `approved_amount`. Your backend must either:

- add a second payment leg for the remainder, or
- cancel the open operation if the sale does not proceed

Do not overwrite the original approval amount during retries.

## Tips and gratuity

Use `tip_amount` on the operation when the tip is part of the fiscalized sale. If your payment processor returns separate payment references for base amount and tip, keep separate payment entries and distinguish them through your own external payment metadata.

## Asynchronous settlement

Some payment methods settle after the operation is fiscally completed. In that case:

- complete the operation with the latest known payment state
- poll the operation or receipt resource for later state changes when needed
- retain the processor and payment references for reconciliation

## Refunds against prior sales

A refund is a new operation with `type: refund`. Each refund payment should include `related_payment_id` and the operation should include `related_operation_id` for the original sale.
