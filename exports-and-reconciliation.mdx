## Overview

Exports are asynchronous. You create an export job, poll it until the artifact is ready, then download the file before the retention deadline.

Every export response includes the schema version, retention deadline, and a download link once the status reaches `ready`.

## Export types

| Type | Description |
| --- | --- |
| `operations` | All POS operations in the time range |
| `receipts` | Receipt data |
| `audit_events` | Audit trail events |
| `dsfinvk` | Germany (KassenSichV) — DSFinV-K archive containing 20 CSV files, `index.xml`, and GDPdU DTD |
| `tse_tar` | Germany (KassenSichV) — Raw TSE transaction data |
| `dep7` | Austria (RKSV) — DEP7 data export |

The first three are generic data exports for ERP sync, lake ingestion, and reconciliation. The last three are country-specific fiscal exports scoped to a single register.

```json
{
  "type": "operations",
  "from": "2026-03-01T00:00:00Z",
  "to": "2026-03-31T23:59:59Z"
}
```

## Fiscal exports

Fiscal exports produce compliance artifacts required by local tax authorities. They require a `register_id`.

<Tabs>
  <Tab title="Germany — DSFinV-K">
    ```bash
    curl -X POST https://sandbox.api.openfiskal.com/v1/exports \
      -H "Authorization: Bearer ofk_test_abc123" \
      -H "X-OpenFiskal-Organization: org_01HXYZ" \
      -H "Idempotency-Key: dsfinvk-2026-03" \
      -H "Content-Type: application/json" \
      -d '{
        "type": "dsfinvk",
        "register_id": "reg_01HABC",
        "from": "2026-03-01T00:00:00Z",
        "to": "2026-03-31T23:59:59Z"
      }'
    ```

    The resulting TAR archive contains the full DSFinV-K dataset: `cashpointclosing.csv`, `transactions.csv`, `lines.csv`, `payment.csv`, `vat.csv`, and other mandated files alongside `index.xml` and the GDPdU DTD.
  </Tab>
  <Tab title="Germany — TSE data">
    ```bash

      -H "Authorization: Bearer ofk_test_abc123" \
    ```

    Returns the raw TSE (Technical Security Equipment) transaction log for the register's signing unit.

### Polling and download

All export types follow the same async lifecycle:

1. `POST /exports` returns a job with status `queued`
2. Poll `GET /exports/{exportId}` until status is `ready`

```json
{
  "id": "exp_01JDEF",
  "type": "dsfinvk",
  "status": "ready",
  "schema_version": "1.0",
  "register_id": "reg_01HABC",
  "from": "2026-03-01T00:00:00Z",
  "to": "2026-03-31T23:59:59Z",
  "retention_expires_at": "2026-04-07T00:00:00Z",
  "created_at": "2026-03-31T12:00:00Z"
}
```

## Reconciliation guidance

Reconcile by:

- operation ID
- external ID
- payment ID
- receipt ID

Use exports for bulk comparison. Poll export and operation resources when you need updated state.

## Retention

Export artifacts remain downloadable until the `retention_expires_at` timestamp. Copy each artifact to your own durable storage before that deadline.
