title: "Getting started"

This guide walks you through the current `v1` integration contract. You create an API key, onboard a merchant, register a location and terminal, start an operation, update it with `If-Match`, and complete it with the typed payment contract.

<Info>
  All examples use `https://sandbox.api.openfiskal.com/v1`. Replace with `https://api.openfiskal.com/v1` when you go live.
</Info>

## Prerequisites

- An OpenFiskal tenant
- A tenant-scoped, country-scoped API key created in your tenant dashboard
- `curl` or an HTTP client
- A backend service / database to store your API key, entity IDs and ETags.

## Authentication model

Use the standard bearer header on every request:

```bash
-H "Authorization: Bearer ofk_live_abc123..."
```

Every organization-scoped request must also include:

```bash
-H "X-OpenFiskal-Organization: org_01HXYZ"
```

This header value is the organization ID returned by `POST /organizations`, not your API key.

## Step by step

<Steps>
  <Step title="Create your API key">
    Start in the OpenFiskal tenant dashboard. Create a tenant-scoped, country-scoped API key for the country you want to operate in, then store it in your secrets manager.
  </Step>
  <Step title="Create an organization">
    An organization represents the merchant legal entity.

    ```bash
    curl -X POST https://sandbox.api.openfiskal.com/v1/organizations \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "Content-Type: application/json" \
      -d '{
        "legal_name": "Mustermann GmbH",
        "country_code": "DEU",
        "tax_id": "DE123456789",
        "address": {
          "line1": "Friedrichstraße 42",
          "city": "Berlin",
          "postal_code": "10117",
          "country_code": "DEU"
        }
      }'
    ```

    The returned `id` becomes the `X-OpenFiskal-Organization` header on every organization-scoped request that follows.
  </Step>
  <Step title="Create a location">
    A location is a physical point of sale such as a store or restaurant.

    ```bash
    curl -X POST https://sandbox.api.openfiskal.com/v1/locations \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "X-OpenFiskal-Organization: org_01HXYZ" \
      -H "Content-Type: application/json" \
      -d '{
        "name": "Berlin Mitte",
        "address": {
          "line1": "Friedrichstraße 42",
          "city": "Berlin",
          "postal_code": "10117",
          "country_code": "DEU"
        },
        "timezone": "Europe/Berlin"
      }'
    ```
  </Step>
  <Step title="Create a register">
    A register is the logical checkout or terminal that creates operations.

    ```bash
    curl -X POST https://sandbox.api.openfiskal.com/v1/registers \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "X-OpenFiskal-Organization: org_01HXYZ" \
      -H "Content-Type: application/json" \
      -d '{
        "location_id": "loc_01HXYZ",
        "country_code": "DEU",
        "name": "Register 1",
        "external_id": "pos-register-1"
      }'
    ```
  </Step>
  <Step title="Start an operation">
    Start every operation with `POST /operations`. The request includes `source`, your upstream `external_id`, and a typed operation variant in `data`. Persist the returned `id`, `resource_version`, and `ETag`.

    ```bash
    curl -X POST https://sandbox.api.openfiskal.com/v1/operations \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "X-OpenFiskal-Organization: org_01HXYZ" \
      -H "Idempotency-Key: op-order-1001-start" \
      -H "Content-Type: application/json" \
      -d '{
        "location_id": "loc_01HXYZ",
        "register_id": "reg_01HXYZ",
        "source": "POS",
        "external_id": "order-1001",
        "data": {
          "type": "sale",
          "status": "open",
          "currency": "EUR",
          "amount": 4250,
          "tip_amount": 0
        }
      }'
    ```
  </Step>
  <Step title="Update the operation while it is open">
    You can patch the operation as long as it is still mutable. Send the latest operation `ETag` in `If-Match`. Every successful mutation returns a new `resource_version` and `ETag`, so replace your stored value after each write. If another writer updated the operation first, re-read the resource and retry with the new `ETag`.

    ```bash
    curl -X PATCH https://sandbox.api.openfiskal.com/v1/operations/op_01HXYZ \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "X-OpenFiskal-Organization: org_01HXYZ" \
      -H 'If-Match: "opv_01HXYZ"' \
      -H "Content-Type: application/json" \
      -d '{
        "data": {
          "type": "sale",
          "status": "pending_payment",
          "currency": "EUR",
          "amount": 4750,
          "line_items": [
            {
              "name": "Lunch menu",
              "quantity": 1,
              "unit_price": 4750,
              "total_amount": 4750
            }
          ]
        }
      }'
    ```
  </Step>
  <Step title="Complete the operation">
    Completion includes typed payment legs. Use one entry per tender leg and include processor references where available. The completed operation can also return `fiscal_information` alongside the newer fiscal document shape. Store the returned `ETag` from the completion response if you will read the final operation again later.

    ```bash
    curl -X PATCH https://sandbox.api.openfiskal.com/v1/operations/op_01HXYZ/complete \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "X-OpenFiskal-Organization: org_01HXYZ" \
      -H "Idempotency-Key: op-order-1001-complete" \
      -H 'If-Match: "opv_01HXYA"' \
      -H "Content-Type: application/json" \
      -d '{
        "payments": [
          {
            "payment_id": "pay_1001_card",
            "method": "card",
            "kind": "sale",
            "amount": 4750,
            "currency": "EUR",
            "status": "captured",
            "processor": "shopify_payments",
            "card_brand": "visa",
            "processor_reference": "ch_123",
            "processed_at": "2026-02-26T12:05:00Z"
          }
        ]
      }'
    ```
  </Step>
  <Step title="Read and list operations">
    Use `GET /operations` to list operations and `GET /operations/{id}` to read one operation, including its current `resource_version`.

    ```bash
    curl https://sandbox.api.openfiskal.com/v1/operations \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "X-OpenFiskal-Organization: org_01HXYZ"
    ```
  </Step>
  <Step title="Create an export job">
    Exports are asynchronous. Create a time-bounded export job and poll it until it is ready.

    ```bash
    curl -X POST https://sandbox.api.openfiskal.com/v1/exports \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "X-OpenFiskal-Organization: org_01HXYZ" \
      -H "Content-Type: application/json" \
      -d '{
        "type": "operations",
        "format": "jsonl",
        "from": "2026-02-01T00:00:00Z",
        "to": "2026-02-29T23:59:59Z"
      }'
    ```
  </Step>
</Steps>
