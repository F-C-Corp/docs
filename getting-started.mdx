---
title: "Getting Started"
description: "Create your API key, onboard a merchant, and complete your first operation"
---

This guide walks you through the smallest useful OpenFiskal integration. You create an API key in the tenant dashboard, onboard a merchant, create its location and register, trigger fiscalization for that register, and then process operations.

<Info>
  All examples use `https://sandbox.api.openfiskal.com/v1`. Replace with `https://api.openfiskal.com/v1` when you go live.
</Info>

## Prerequisites

- An OpenFiskal tenant
- A tenant-scoped, country-scoped API key created in your tenant dashboard
- `curl` or an HTTP client

## Authentication model

Use the standard bearer header on every request:

```bash
-H "Authorization: Bearer ofk_live_abc123..."
```

Every organization-scoped request must also include:

```bash
-H "X-OpenFiskal-Organization: org_01HXYZ"
```

You only omit `X-OpenFiskal-Organization` when you create, update, or delete the organization itself.

## Step by step

<Steps>
  <Step title="Create your API key">
    Start in the OpenFiskal tenant dashboard. Create a tenant-scoped, country-scoped API key for the country you want to operate in, then store it in your secrets manager.
  </Step>
  <Step title="Create an organization">
    An organization represents the merchant legal entity.

    ```bash
    curl -X POST https://sandbox.api.openfiskal.com/v1/organizations \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "Content-Type: application/json" \
      -d '{
        "legal_name": "Mustermann GmbH",
        "country": "DE",
        "tax_id": "DE123456789",
        "address": {
          "line1": "Friedrichstraße 42",
          "city": "Berlin",
          "postal_code": "10117",
          "country": "DE"
        }
      }'
    ```

    ```json
    {
      "id": "org_01HXYZ",
      "legal_name": "Mustermann GmbH",
      "country": "DE",
      "tax_id": "DE123456789",
      "created_at": "2026-02-26T10:01:00Z",
      "updated_at": "2026-02-26T10:01:00Z"
    }
    ```

    Note the returned `id`. Use it as the `X-OpenFiskal-Organization` header on every organization-scoped request that follows.
  </Step>
  <Step title="Create a location">
    A location is a physical point of sale such as a store, restaurant, or kiosk.

    ```bash
    curl -X POST https://sandbox.api.openfiskal.com/v1/locations \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "X-OpenFiskal-Organization: org_01HXYZ" \
      -H "Content-Type: application/json" \
      -d '{
        "name": "Berlin Mitte",
        "address": {
          "line1": "Friedrichstraße 42",
          "city": "Berlin",
          "postal_code": "10117",
          "country": "DE"
        },
        "timezone": "Europe/Berlin"
      }'
    ```

    ```json
    {
      "id": "loc_01HXYZ",
      "organization_id": "org_01HXYZ",
      "name": "Berlin Mitte",
      "timezone": "Europe/Berlin",
      "created_at": "2026-02-26T10:02:00Z",
      "updated_at": "2026-02-26T10:02:00Z"
    }
    ```
  </Step>
  <Step title="Create a register">
    A register is the device or logical checkout that will create operations.

    ```bash
    curl -X POST https://sandbox.api.openfiskal.com/v1/registers \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "X-OpenFiskal-Organization: org_01HXYZ" \
      -H "Content-Type: application/json" \
      -d '{
        "location_id": "loc_01HXYZ",
        "name": "Register 1",
        "external_id": "pos-register-1"
      }'
    ```

    ```json
    {
      "id": "reg_01HXYZ",
      "organization_id": "org_01HXYZ",
      "location_id": "loc_01HXYZ",
      "name": "Register 1",
      "external_id": "pos-register-1",
      "status": "active",
      "created_at": "2026-02-26T10:03:00Z",
      "updated_at": "2026-02-26T10:03:00Z"
    }
    ```
  </Step>
  <Step title="Trigger fiscalization for the register">
    Fiscalization is modeled as a register sub-resource so the target register is explicit.

    ```bash
    curl -X POST https://sandbox.api.openfiskal.com/v1/registers/reg_01HXYZ/fiscalizations \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "X-OpenFiskal-Organization: org_01HXYZ"
    ```

    ```json
    {
      "id": "rfj_01HXYZ",
      "register_id": "reg_01HXYZ",
      "status": "queued",
      "created_at": "2026-02-26T10:04:00Z"
    }
    ```
  </Step>
  <Step title="Start an operation">
    Start every operation with `POST /operations`. The request includes the source and a typed `data` object.

    ```bash
    curl -X POST https://sandbox.api.openfiskal.com/v1/operations \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "X-OpenFiskal-Organization: org_01HXYZ" \
      -H "Idempotency-Key: op-order-1001-start" \
      -H "Content-Type: application/json" \
      -d '{
        "location_id": "loc_01HXYZ",
        "register_id": "reg_01HXYZ",
        "source": "POS",
        "data": {
          "type": "sale",
          "external_reference": "order-1001",
          "currency": "EUR",
          "amount": 4250
        }
      }'
    ```

    ```json
    {
      "operation_id": "op_01HXYZ",
      "status": "open"
    }
    ```
  </Step>
  <Step title="Update the operation while it is open">
    You can patch the operation as long as it is not completed yet.

    ```bash
    curl -X PATCH https://sandbox.api.openfiskal.com/v1/operations/op_01HXYZ \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "X-OpenFiskal-Organization: org_01HXYZ" \
      -H "Content-Type: application/json" \
      -d '{
        "data": {
          "type": "sale",
          "external_reference": "order-1001",
          "currency": "EUR",
          "amount": 4750,
          "line_items": [
            {
              "name": "Lunch menu",
              "quantity": 1,
              "unit_price": 4750,
              "total_amount": 4750
            }
          ]
        }
      }'
    ```
  </Step>
  <Step title="Complete the operation">
    Completion is modeled as an operation sub-resource and includes the reconciled payment transactions.

    ```bash
    curl -X PATCH https://sandbox.api.openfiskal.com/v1/operations/op_01HXYZ/completion \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "X-OpenFiskal-Organization: org_01HXYZ" \
      -H "Idempotency-Key: op-order-1001-complete" \
      -H "Content-Type: application/json" \
      -d '{
        "transactions": [
          {
            "transaction_id": "txn_1001_1",
            "payment_type": "card",
            "kind": "sale",
            "amount": 4750,
            "currency": "EUR",
            "status": "success",
            "gateway": "shopify_payments",
            "card_brand": "visa",
            "external_reference": "ch_123",
            "processed_at": "2026-02-26T12:05:00Z"
          }
        ]
      }'
    ```

    ```json
    {
      "operation_id": "op_01HXYZ",
      "status": "completed",
      "fiscal_information": {
        "fiscalization_id": "fisc_01HXYZ",
        "document_number": "2026-000123",
        "qr_code_data": "OFK:2026-000123:...",
        "receipt_url": "https://sandbox.api.openfiskal.com/v1/receipts/rcp_01HXYZ",
        "fiscalized_at": "2026-02-26T12:05:01Z"
      }
    }
    ```
  </Step>
  <Step title="Read and list operations">
    Use `GET /operations` to list operations and `GET /operations/:id` to read one operation.

    ```bash
    curl https://sandbox.api.openfiskal.com/v1/operations \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "X-OpenFiskal-Organization: org_01HXYZ"
    ```
  </Step>
  <Step title="Create an export job">
    Exports are asynchronous. First create the job.

    ```bash
    curl -X POST https://sandbox.api.openfiskal.com/v1/exports \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "X-OpenFiskal-Organization: org_01HXYZ" \
      -H "Content-Type: application/json" \
      -d '{
        "type": "operations",
        "format": "csv",
        "from": "2026-02-01T00:00:00Z",
        "to": "2026-02-29T23:59:59Z"
      }'
    ```

    ```json
    {
      "id": "exp_01HXYZ",
      "status": "queued",
      "type": "operations",
      "format": "csv",
      "created_at": "2026-02-26T12:10:00Z",
      "completed_at": null,
      "download_url": null
    }
    ```
  </Step>
  <Step title="Check export status and download">
    Read the export job first. Download only after it is completed.

    ```bash
    curl https://sandbox.api.openfiskal.com/v1/exports/exp_01HXYZ \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "X-OpenFiskal-Organization: org_01HXYZ"
    ```

    ```bash
    curl -L https://sandbox.api.openfiskal.com/v1/exports/exp_01HXYZ/download \
      -H "Authorization: Bearer ofk_live_abc123..." \
      -H "X-OpenFiskal-Organization: org_01HXYZ" \
      -o export.csv
    ```
  </Step>
</Steps>

## Final endpoint set

The documentation is now centered on this API surface:

- `POST /organizations`
- `PATCH /organizations/:id`
- `DELETE /organizations/:id`
- `POST /locations`
- `GET /locations`
- `GET /locations/:id`
- `PATCH /locations/:id`
- `DELETE /locations/:id`
- `POST /registers`
- `GET /registers`
- `GET /registers/:id`
- `PATCH /registers/:id`
- `DELETE /registers/:id`
- `POST /registers/:id/fiscalizations`
- `POST /operations`
- `GET /operations`
- `GET /operations/:id`
- `PATCH /operations/:id`
- `PATCH /operations/:id/completion`
- `POST /exports`
- `GET /exports/:id`
- `GET /exports/:id/download`

## Why this shape is better

- Auth uses the standard `Authorization` header instead of a custom API key header.
- Fiscalization targets a concrete register resource.
- Completion is modeled as an operation sub-resource instead of a verb endpoint.
- Export status and export download are separate, which is easier for asynchronous clients.
