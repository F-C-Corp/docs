---
title: "Decimal precision"
description: "How OpenFiskal validates monetary decimal values, the maximum precision accepted on the wire, and which derived fields are checked at request time."
---

OpenFiskal stores and computes every monetary amount as an arbitrary-precision decimal — never as a `float`. Integrators send amounts as **decimal strings** (for example, `"100.00"`, `"-19.00"`, `"0.19"`) so values round-trip exactly across systems with different numeric representations.

This page documents two boundaries you can rely on:

1. The **maximum precision** the API accepts for a decimal field, and
2. The **derived-field checks** that run at request time so a payload with internally inconsistent amounts is rejected before it can be persisted or fiscalized.

## Maximum precision: 8 fractional digits

Every decimal-string field on the API contract is capped at **8 digits after the decimal point**. Submitting a value with more fractional digits is rejected with `decimal_precision_exceeded` (HTTP `422`). The cap applies regardless of whether the field would normally be expressed in cents (`"15.97"`), fractional cents (`"15.96638655"`), or as a tax rate (`"0.19"`).

Why 8? It is enough to express any legally settled monetary amount in any of the currencies OpenFiskal supports, while still fitting comfortably in fixed-point arithmetic on every implementation we interoperate with (databases, fiscal signing devices, downstream exports). Integrators that compute at higher precision must round before submitting.

## Derived-field checks

When a field can be derived from other fields on the same payload (for example, `tax_amount` from `total_amount` and `rate`), OpenFiskal recomputes the expected value and rejects the request if your value does not match.

### `line_items[].taxes[].tax_amount`

For every `taxes` entry on a goods-movement line item (`SALE`, `RETURN`, `EXCHANGE`), we validate that the submitted `tax_amount` is consistent with the line's `total_amount` (gross, tax-inclusive) and the `rate` on the same tax row.

The expected value is computed using the inclusive-tax formula:

```
expected_tax = total_amount × rate / (1 + rate)
```

Comparison is **strict at 8 decimal places**. The expected value is rounded half-up to 8 dp, and your submitted `tax_amount` is taken at face value (a submission of `"15.97"` is `15.97000000` to the validator — it is **not** treated as a 2-dp rounding of some more precise underlying value). Cent-rounded submissions that don't equal the 8-dp expected value are rejected with `tax_amount_precision_invalid` (HTTP `422`).

Practically, this means integrators must submit `tax_amount` at the precision the math actually produces. For a line whose gross and rate yield an exact value (for example, `total_amount: "119.00"`, `rate: "0.19"` → exactly `19.00`), submitting `"19.00"` is fine. For a line whose math produces a recurring decimal (for example, `total_amount: "100.00"`, `rate: "0.19"` → `15.96638655462184874...`), submit the value rounded to 8 dp (`"15.96638655"`) — not `"15.97"`.

Examples:

| `total_amount` | `rate` | Submitted `tax_amount` | Expected at 8 dp | Result |
| --- | --- | --- | --- | --- |
| `"119.00"` | `"0.19"` | `"19.00"` | `19.00000000` | ✅ accepted (math is exact) |
| `"100.00"` | `"0.19"` | `"15.96638655"` | `15.96638655` | ✅ accepted |
| `"100.00"` | `"0.19"` | `"15.97"` | `15.96638655` | ❌ rejected (face value `15.97000000` ≠ expected) |
| `"100.00"` | `"0.19"` | `"19.00"` | `15.96638655` | ❌ rejected (exclusive-math result) |
| `"100.00"` | `"0.19"` | `"15.96638656"` | `15.96638655` | ❌ rejected (off by 1 ULP at 8 dp) |
| `"-100.00"` | `"0.19"` | `"-15.96638655"` | `-15.96638655` | ✅ accepted (`RETURN` sign convention) |

#### Multi-tax line items

A line item that carries more than one tax entry (for example, US-style state + municipal stacking) is **not** subject to the per-tax precision check at this layer — the formula relating each individual `tax_amount` to the line's gross is jurisdiction-specific. The operation-level identity (`pretax_amount + tax_amount + tip_amount = total_amount`) and the country-specific exporter rules continue to apply. Today OpenFiskal's fiscal exporters (KassenSichV, RKSV, RT) require single-VAT-per-line-item, so multi-tax lines are rejected later in the pipeline if they reach a regime that disallows them.

### Operation totals identity

The operation-level totals are checked exactly: `pretax_amount + tax_amount + tip_amount` must equal `total_amount`. This identity is enforced strictly (no rounding) for every `SALE`, `RETURN`, and `EXCHANGE` request body. A mismatch returns `validation_error` (HTTP `422`) with the offending sum in the error message.

## Sign conventions are independent

Decimal precision validation does not change the sign rules. `RETURN` operations carry negative amounts, `SALE` operations carry non-negative amounts, and `EXCHANGE` operations may net positive or negative. The precision check applies symmetrically to negative values: `-100.00 × 0.19 / 1.19 = -15.96638655462184874...`, rounded half-up at 8 dp to `-15.96638655`. See [Transaction lifecycle](/pos-operation-ingestion) for the full sign rules.

Sign violations (for example, a negative `total_amount` on a `SALE`) are reported before the precision check fires, so the more fundamental error message surfaces first.

## Future expansions

Other derivable decimal fields (for example, `line_items[].total_amount = quantity × unit_price`, or operation-level reconciliation against the sum of per-line tax amounts) may gain similar request-time precision checks in future API versions. Such additions will be announced in the [Changelog](/changelog) before enforcement lands.
