---
title: "Decimal precision"
description: "How OpenFiskal validates monetary decimal values, the maximum precision accepted on the wire, and which derived fields are checked at request time."
---

OpenFiskal stores and computes every monetary amount as an arbitrary-precision decimal — never as a `float`. Integrators send amounts as **decimal strings** (for example, `"100.00"`, `"-15.97"`, `"0.19"`) so values round-trip exactly across systems with different numeric representations.

This page documents two boundaries you can rely on:

1. The **maximum precision** the API accepts for a decimal field, and
2. The **derived-field checks** that run at request time so a payload with internally inconsistent amounts is rejected before it can be persisted or fiscalized.

## Maximum precision: 8 fractional digits

Every decimal-string field on the API contract is capped at **8 digits after the decimal point**. Submitting a value with more fractional digits is rejected with [`decimal_precision_exceeded`](/errors#validation-errors-422) (HTTP `422`). The cap applies regardless of whether the field would normally be expressed in cents (`"15.97"`), fractional cents (`"15.96638655"`), or as a tax rate (`"0.19"`).

Why 8? It is more than enough to express any legally settled monetary amount in any of the currencies OpenFiskal supports, while still fitting comfortably in 64-bit fixed-point arithmetic on every implementation we interoperate with (databases, fiscal signing devices, downstream exports). Integrators that compute at higher precision must round before submitting; we round half-up to match the precision you provided when we run derived-field checks (below), so submitting a value at 2 decimal places does not penalise you against a more precise expected value.

If you are choosing a precision for your own integration, send what you compute. There is no requirement that values be cent-rounded — `"15.96638655"` is just as valid as `"15.97"` for a tax amount, and we evaluate them at the precision you sent.

## Derived-field checks

When a field can be derived from other fields on the same payload (for example, `tax_amount` from `total_amount` and `rate`), OpenFiskal recomputes the expected value and rejects the request if your value does not match — at the precision you submitted.

### `line_items[].taxes[].tax_amount`

For every `taxes` entry on a goods-movement line item (`SALE`, `RETURN`, `EXCHANGE`), we validate that the submitted `tax_amount` is consistent with the line's `total_amount` (gross, tax-inclusive) and the `rate` on the same tax row.

The expected value is computed using the inclusive-tax formula:

```
expected_tax = total_amount × rate / (1 + rate)
```

Both the expected value and the submitted value are then rounded **half-up** to the number of decimal places present in your `tax_amount` string, and compared for equality. If they differ, the request is rejected with [`tax_amount_precision_invalid`](/errors#validation-errors-422) (HTTP `422`).

Examples:

| `total_amount` | `rate` | Submitted `tax_amount` | Expected (HALF_UP at submitted dp) | Result |
| --- | --- | --- | --- | --- |
| `"100.00"` | `"0.19"` | `"15.97"` | `15.97` (2 dp) | ✅ accepted |
| `"100.00"` | `"0.19"` | `"15.96638655"` | `15.96638655` (8 dp) | ✅ accepted |
| `"100.00"` | `"0.19"` | `"19.00"` | `15.97` (2 dp) | ❌ rejected |
| `"100.00"` | `"0.19"` | `"15.96638656"` | `15.96638655` (8 dp) | ❌ rejected (off by 1 ULP at 8 dp) |
| `"-100.00"` | `"0.19"` | `"-15.97"` | `-15.97` (2 dp) | ✅ accepted (`RETURN` sign convention) |

#### Multi-tax line items

A line item that carries more than one tax entry (for example, US-style state + municipal stacking) is **not** subject to the per-tax precision check at this layer — the formula relating each individual `tax_amount` to the line's gross is jurisdiction-specific. The operation-level identity (`pretax_amount + tax_amount + tip_amount = total_amount`) and the country-specific exporter rules continue to apply. Today OpenFiskal's fiscal exporters (KassenSichV, RKSV, RT) require single-VAT-per-line-item, so multi-tax lines are rejected later in the pipeline if they reach a regime that disallows them.

### Operation totals identity

The operation-level totals are checked exactly: `pretax_amount + tax_amount + tip_amount` must equal `total_amount`. This identity is enforced strictly (no rounding) for every `SALE`, `RETURN`, and `EXCHANGE` request body. A mismatch returns `validation_error` (HTTP `422`) with the offending sum in the error message.

## Sign conventions are independent

Decimal precision validation does not change the sign rules. `RETURN` operations carry negative amounts, `SALE` operations carry non-negative amounts, and `EXCHANGE` operations may net positive or negative. The precision check applies symmetrically to negative values: `-100.00 × 0.19 / 1.19 = -15.96638...`, rounded half-up at 2 decimal places to `-15.97`. See [Transaction lifecycle](/pos-operation-ingestion) for the full sign rules.

## Future expansions

Other derivable decimal fields (for example, `line_items[].total_amount = quantity × unit_price`, or operation-level reconciliation against the sum of per-line tax amounts) may gain similar request-time precision checks in future API versions. Such additions will be announced in the [Changelog](/changelog) before enforcement lands.
